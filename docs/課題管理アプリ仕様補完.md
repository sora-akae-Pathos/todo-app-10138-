# 課題管理アプリ仕様補完

## 1. 認証・アカウント（サインイン／サインアップ／ログアウト）

| 項目                   | 決定内容                                                                                                                                                              |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 認証方式               | メールアドレスとパスワードによる本人確認（ソーシャルログインは行わない）。                                                                                            |
| ユーザープロファイル   | 必須: メールアドレス（一意）、パスワード、表示名。表示名は UI 上のユーザー表記および「最終更新者」表示に用いる。                                                      |
| 表示名の制約           | 1〜30 文字。前後空白は保存時にトリム。改行不可。                                                                                                                      |
| メール                 | 重複登録はエラー表示。                                                                                                                                                |
| パスワード             | 8〜128 文字。                                                                                                                                                         |
| メール確認             | 行わない。登録直後からサインイン可能。 ただし、墓の実装もすべて完成して余裕があれば、メール確認を行い、認証された人、メールアドレスのみログイン可能とする形としたい。 |
| セッション             | 認証プロバイダの既定セッションに従う。複数端末からの同時利用を許可せず、一台に制限する。                                                                              |
| ログアウト             | クライアント上の認証状態を破棄し、サインイン画面へ遷移。サーバー側トークン失効はプロバイダの標準動作に従う。                                                          |
| サインアップ直後の遷移 | ホーム画面へ遷移。                                                                                                                                                    |
| サインイン直後の遷移   | 直前に開こうとしていた保護ルートがあればそこへ、なければホーム。                                                                                                      |

## 2. プロジェクト（作成・検索・参加・合言）

| 項目                                     | 決定内容                                                                                                                             |
| ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| 合言の意味                               | プロジェクトへの参加時に入力し、作成時に設定した値と一致すれば当該ユーザーをプロジェクトメンバーとする共有秘密。                     |
| 作成時入力                               | プロジェクト名（必須）、合言（必須）。カタログ通り。                                                                                 |
| 合言の形式                               | 8〜32 文字。軽い参加制御のためそのまま保存する。ただし、将来ハッシュ化も使う可能性があるため、拡張しやすい構成とすること。           |
| プロジェクト名                           | 1〜80 文字。前後空白はトリム。同一ユーザーが同一プロジェクト名を重複作成することは許可する（グローバル一意は課さない）。             |
| 検索対象                                 | プロジェクト名のみ。部分一致。大文字小文字は区別しない。                                                                             |
| 検索結果                                 | 最大 50 件。更新日時の降順で並べる。0 件のときは「該当するプロジェクトがありません」等のメッセージを表示。                           |
| 参加フロー                               | 検索結果からプロジェクトを選び、合言を入力して参加を確定。一致しなければエラーメッセージを表示。試行回数のロックアウトは10回とする。 |
| ユーザーとプロジェクト                   | 多対多。中間データに「参加日時」を保持する。                                                                                         |
| プロジェクトからの脱退・プロジェクト削除 | カタログにないため本アプリの必須スコープには含めない（将来拡張としてもよいが初期実装では未対応でよい）。                             |
| メンバー権限                             | カタログ通り。参加ユーザーは課題およびプロジェクト内データの閲覧・編集が可能（ロール分離なし）。                                     |

## 3. 課題データモデル（必須・デフォルト・文字数・期限）

| 項目               | 決定内容                                                                                                                  |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------- |
| 担当者             | 最大 1 人。未選択を許可（必須はタイトル・期限のみに合わせる）。選択肢は当該プロジェクトのメンバー一覧。                   |
| 優先度の初期値     | 「中」。                                                                                                                  |
| ステータスの初期値 | 「未対応」。                                                                                                              |
| 「無期」と期限     | 「無期」は優先度の取りうる値であり、期限日の必須とは独立する。期限は常に 1 日を指定する必須フィールドとする（日付のみ）。 |
| 期限の粒度         | 日付のみ（カレンダー 1 日単位）。解釈および「今日」「明日」の判定はすべて Asia/Tokyo の日付境界（0:00〜23:59）で行う。    |
| 詳細な内容         | 最大 500 文字。改行可。                                                                                                   |
| 対応方法・完了条件 | いずれも最大 100 文字。改行可。未入力可。                                                                                 |
| 作成日時・更新日時 | サーバー時刻で記録。                                                                                                      |
| 最終更新者         | 課題の作成・更新がサーバーで成功するたびに、当該操作を行ったユーザー ID（および表示用に表示名）を記録する。               |

## 4. 画面構成・遷移・レイアウト

| 項目                                                                                               | 決定内容                                                                                                                                                                                    |
| -------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 画面一覧                                                                                           | カタログの 6 画面（サインイン、サインアップ、プロジェクト詳細、課題作成・編集、課題詳細、ホーム）に準拠。                                                                                   |
| 課題作成・編集                                                                                     | 同一種別の画面として実装してよい。新規と編集で URL を分けるか同一 URL でモード切替かは実装任意だが、ユーザーがブックマーク可能な単位で区切ることを推奨。                                    |
| 作成キャンセル                                                                                     | 一覧（プロジェクト詳細）へ戻る。未送信の入力は「下書き保持」節のルールに従う（破棄しない）が、キャンセルされたものは保持しなくてよい。                                                      |
| 課題詳細                                                                                           | 課題の全項目を表示（読み取り専用）。課題を選択し、「編集」操作で課題作成・編集画面へ遷移。                                                                                                  |
| プロジェクト詳細のグループ順                                                                       | 画面上のセクション順: 未対応 → 対応中 → 苦戦中 → 保留 → 完了。                                                                                                                              |
| グループ内の並び                                                                                   | カタログのデフォルトソート（期限昇順、同一期限は作成日時昇順）を各ステータスグループ内に適用する。                                                                                          |
| ユーザーによるソート変更時                                                                         | ステータスによるグルーピングは維持し、変更したソートキーは各グループ内の並びにのみ適用する（フラット化しない）。                                                                            |
| ホームのレイアウト                                                                                 | プロジェクトと今日明日が期限の課題は明確に分ける。「期限が今日または明日かつ担当者が自分かつステータスが完了以外」の課題のみをリスト表示。課題と参加プロジェクト一覧は縦に2列の配置とする。 |
| ホームで参加プロジェクトがない場合は、プロジェクトへの参加か、プロジェクトの作成を促す文言を表示。 |
| プロジェクト詳細画面で課題が一つも作成されていない場合は、課題のの作成を促す文言を表示。           |

## 5. フィルタ・検索・ソート（併用ルールの具体化）

| 項目                       | 決定内容                                                                                          |
| -------------------------- | ------------------------------------------------------------------------------------------------- |
| 併用時の優先               | カタログ通り: フィルタ → 検索 → ソート。                                                          |
| フィルタの論理             | 複数のフィルタ条件はすべて AND。                                                                  |
| ステータス・優先度フィルタ | いずれか 1 つ以上を選択可能とし、同一軸内は OR（例: 未対応または対応中）、異なる軸同士は AND。    |
| 担当者フィルタ             | 1 人を指定可能。未指定はフィルタなし。                                                            |
| 期限フィルタ               | 開始日・終了日を指定する範囲フィルタ。未指定側は制限なし。                                        |
| 検索（タイトル）           | 部分一致。検索語の前後空白はトリム。大文字小文字は区別しない。全角半角の正規化は行わない（MVP）。 |
| ソートキー                 | カタログの 4 種のみ。単一キー。                                                                   |

## 6. 同時編集・エラー表示

| 項目             | 決定内容                                                                                                                                                                                                                          |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 同時編集         | カタログ通り考慮しない。サーバーは最後に成功した更新を有効とする（last-write-wins）。クライアント側のバージョン競合 UI は不要。                                                                                                   |
| 取得・保存失敗   | カタログ通りエラーメッセージを表示（例: 「データの取得に失敗しました」「保存に失敗しました」）。                                                                                                                                  |
| 更新・削除未完了 | カタログ通り「処理が完了していない旨」を別メッセージで表示（例: 「更新が完了していません」「削除が完了していません」）。技術的には HTTP の 4xx/5xx やネットワークエラーを区別してもよいが、ユーザー向け文言は上記の趣旨を満たす。 |

---

## 7. リロード時の入力保持（短時間の具体化）

| 項目     | 決定内容                                                                                                               |
| -------- | ---------------------------------------------------------------------------------------------------------------------- |
| 対象     | 課題の作成・編集フォームに入力した内容のみ。サインアップ、サインイン、プロジェクト作成、一覧上のフィルタ状態は対象外。 |
| 保持場所 | `sessionStorage`（タブ単位）。                                                                                         |
| 有効期限 | 最終入力から 30 分。超過後は復元しない（破棄）。                                                                       |
| キー単位 | プロジェクト ID と課題 ID（新規は `new` 等の仮 ID）の組でスコープする。                                                |
| 機微情報 | 合言・パスワードを下書き保持の対象に含めない。                                                                         |

---

## 8. オフライン・DB 前提

| 項目         | 決定内容                                                                                                                              |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------- |
| オフライン   | カタログ通り利用不可。初回ロード時または API 失敗時に「ネットワークに接続してから再度お試しください」に相当するメッセージを表示する。 |
| ローディング | データ取得・保存の非同期中は、当該画面または当該操作単位でローディング状態を表示する（カタログの「表示する」に合致すればよい）。      |

---

## 9. XSS 対策（テンプレートバインディングの具体化）

| 項目           | 決定内容                                                                                                                   |
| -------------- | -------------------------------------------------------------------------------------------------------------------------- |
| 方針           | ユーザー入力を `innerHTML` や文字列連結で DOM に埋め込まない。 UI フレームワークの既定のテキストバインディングで表示する。 |
| リッチテキスト | 課題本文を HTML/Markdown として解釈しない（プレーンテキストとして表示）。                                                  |

---

## 10. 非機能（一覧・インフラの最小決定）

| 項目       | 決定内容                                                                                                                                                                                                                   |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 形態       | Web アプリ（ブラウザ）。モバイル専用クライアントは対象外。                                                                                                                                                                 |
| 一覧の分割 | プロジェクト詳細の課題一覧は、ステータスごとに分類されて表示されている。ステータスごとに折りたたみ非表示にすることができる。ページング処理等はせずに生まれた課題はどんどん追加されていき、スクロールで対応することとなる。 |
